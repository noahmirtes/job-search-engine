JOB SCORING SYSTEM DESIGN (HYBRID: DETERMINISTIC + LLM FEATURE EXTRACTION)

GOAL:
Build a robust, flexible, and mostly deterministic job scoring system that:
- Ranks and filters jobs effectively
- Is configurable per user
- Avoids brittle keyword-only logic
- Avoids non-deterministic “LLM decides everything” behavior


HIGH-LEVEL ARCHITECTURE:

                ┌────────────────────┐
                │   Raw Job Posting  │
                │ (title, desc, etc) │
                └─────────┬──────────┘
                          │
        ┌─────────────────┼─────────────────┐
        │                 │                 │
        ▼                 ▼                 ▼

┌────────────────┐  ┌────────────────────┐  ┌────────────────────┐
│ Keyword/Regex  │  │  LLM Classification│  │ Embedding Similarity│
│   Extraction   │  │ (Semantic Features)│  │   (Optional Signal) │
└──────┬─────────┘  └──────────┬─────────┘  └──────────┬─────────┘
       │                        │                       │
       └──────────────┬─────────┴──────────────┬────────┘
                      ▼                        ▼

              ┌────────────────────────────────────┐
              │   Deterministic Scoring Engine     │
              │  (Config-Driven Rule Evaluation)   │
              └────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │ Final Job Score │
                    │  + Explanation  │
                    └─────────────────┘



CORE PRINCIPLES:

1. LLM IS NOT THE SCORER
   - LLM does NOT assign scores
   - LLM ONLY answers structured questions (feature extraction)

2. SCORING IS FULLY DETERMINISTIC
   - All scoring logic lives in config (CSV/YAML/etc)
   - Same inputs → same outputs (given same LLM outputs)

3. FEATURES > OPINIONS
   - LLM produces boolean/enum features
   - Scoring engine converts features → points

4. MULTI-SIGNAL SYSTEM
   - Combine:
     - Keyword matches
     - LLM semantic classifications
     - Embedding similarity (optional)



LLM ROLE (SEMANTIC CLASSIFIER):

Single call per job → structured JSON output

Example:

{
  "junior_position": true,
  "senior_position": false,
  "backend_role": true,
  "python_role": true,
  "frontend_heavy": false,
  "devops_heavy": false,
  "unclear_fields": []
}

Rules:
- Narrow, well-defined questions
- Prefer TRUE / FALSE / UNCLEAR (not free text)
- No scoring logic inside the model
- Model is treated as a feature extractor



SCORING CONFIG (USER-DEFINED):

Each user defines their own scoring rules.

Example (conceptual):

- name: junior_semantic
  type: llm_boolean
  field: junior_position
  score: +30

- name: senior_penalty
  type: llm_boolean
  field: senior_position
  score: -40

- name: python_keyword
  type: keyword
  field: description
  triggers: ["python", "django", "flask"]
  score: +10

- name: avoid_dotnet
  type: keyword
  field: description
  triggers: [".net", "asp.net"]
  score: -15

All scoring is:
- Explicit
- Transparent
- Editable per user



WHY THIS APPROACH:

KEYWORD-ONLY PROBLEM:
- Misses contextual meaning
- Requires massive trigger lists
- Fails when signals are implicit

LLM-ONLY PROBLEM:
- Non-deterministic
- Hard to debug
- Not reproducible

HYBRID SOLUTION:
- Keywords → precise, deterministic matches
- LLM → contextual understanding
- Scoring → controlled, explainable output



DETERMINISM STRATEGY:

System is "mostly deterministic":
- Scoring logic = fully deterministic
- LLM output = constrained + structured

To improve consistency:
- Use strict JSON schema
- Avoid open-ended prompts
- Cache/store LLM outputs
- Version:
  - model
  - prompt
  - scoring rules



PIPELINE SUMMARY:

1. Ingest job posting
2. Extract keyword/regex features
3. Run LLM once → structured feature output
4. (Optional) compute embedding similarity
5. Apply scoring rules (config-driven)
6. Produce:
   - final score
   - explanation (rule breakdown)



KEY BENEFITS:

- Strong ranking + filtering capability
- Per-user customization
- Explainable results
- Extensible (easy to add new rules)
- Robust against missing keywords
- LLM used safely (narrow scope)
